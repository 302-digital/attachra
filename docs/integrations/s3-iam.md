# S3 least-privilege IAM policy (SR-121-4)

Attachra's S3 storage driver (`internal/core/storage/s3`)
only ever needs to read, write, delete, and stat objects inside a
single bucket. It never lists the bucket, never touches bucket-level
configuration, and never generates or relies on presigned URLs
(SR-121-1) — object bytes always flow through Attachra's own process.

Grant the credentials configured under `storage.s3` (see
`internal/config/config.go`) only the following actions, scoped to
that one bucket.

## IAM policy JSON

Replace `REPLACE_WITH_BUCKET_NAME` with the actual bucket name.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AttachraObjectReadWrite",
      "Effect": "Allow",
      "Action": [
        "s3:PutObject",
        "s3:GetObject",
        "s3:DeleteObject"
      ],
      "Resource": "arn:aws:s3:::REPLACE_WITH_BUCKET_NAME/*"
    }
  ]
}
```

Notes:

- `s3:GetObject` also covers `HeadObject` (used by `Driver.Stat`) —
  AWS IAM does not have a separate action for `HeadObject`; the same
  `s3:GetObject` permission authorizes it.
- No `s3:ListBucket` is granted: Attachra never lists bucket
  contents, only ever addresses objects it already knows the key of
  (SR-121-3, object keys are opaque IDs generated by
  `storage.NewObjectKey` and never derived from a listing).
- No bucket-level actions (`s3:GetBucketLocation`,
  `s3:PutBucketPolicy`, `s3:PutBucketAcl`, etc.) are granted or
  required.
- No `s3:PutObjectAcl` or any ACL-setting action is granted: the
  driver never sets a canned/public ACL on objects (SR-121-1); bucket
  and object ACLs should be managed once, out-of-band, by whoever
  provisions the bucket (see "Bucket-level configuration" below).

## Server-side encryption (SR-121-2)

If `storage.s3.sse` is set to `aws:kms`, the IAM principal (or the
KMS key's own key policy) must additionally allow
`kms:GenerateDataKey` and `kms:Decrypt` on the configured
`storage.s3.sse_kms_key_id`:

```json
{
  "Sid": "AttachraKMSForSSE",
  "Effect": "Allow",
  "Action": [
    "kms:GenerateDataKey",
    "kms:Decrypt"
  ],
  "Resource": "arn:aws:kms:REGION:ACCOUNT_ID:key/REPLACE_WITH_KMS_KEY_ID"
}
```

If `storage.s3.sse` is `AES256` (SSE-S3) or empty, no additional KMS
permission is needed.

## Bucket-level configuration (out of scope for the IAM policy above)

Provisioned once by whoever creates the bucket, not by Attachra at
runtime:

- Bucket policy / ACLs: fully private, no public read/write.
- Block Public Access: enabled (all four settings on).
- Default encryption: recommended to enable at the bucket level in
  addition to (or instead of) the driver's per-request SSE headers.
- Versioning: optional, operator's choice (Attachra does not depend
  on or manage object versions).

## MinIO

MinIO uses its own policy JSON dialect but accepts an
AWS-IAM-compatible policy for bucket-scoped grants; the same actions
list (`s3:PutObject`, `s3:GetObject`, `s3:DeleteObject`, no
`s3:ListBucket`) attached to a dedicated MinIO user/service-account is
the equivalent minimal grant. See `deploy/dev/docker-compose.yml` for
the local dev MinIO instance (bucket `attachra-dev`, credentials
`attachra-dev` / `attachra-dev-secret`) — the dev setup does not
currently scope a dedicated least-privilege MinIO policy for that
bucket; it uses the MinIO root user for simplicity in local
development only. Production deployments should create a
bucket-scoped MinIO user with the policy above instead of using root
credentials.
